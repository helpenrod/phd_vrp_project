## 🧬 Framework Workflow Description

### I. High-Level Goal

The project is a **generative hyper-heuristic framework** for solving Vehicle Routing Problems (VRPs). Its purpose is not to be a single, static algorithm, but rather a system that can **automatically construct a new, specialized algorithm** based on a high-level description of a problem.

It achieves this by separating the generic algorithmic logic (the "engine") from the problem-specific constraints and the library of available operators (the "building blocks").

### II. The Core Components and Their Roles

The entire system is built on a clear separation of concerns, reflected in the directory structure:

1.  **The `core/` Directory (The Engine Room):** This is the heart of the framework. It is completely **problem-agnostic**.
    *   **`ga_framework.py`:** Contains the `GAFramework` class. This is a generic template for a Genetic Algorithm. It knows how to run a GA loop (selection, crossover, mutation, evaluation), but it has no hard-coded knowledge of any specific VRP constraint (like capacity or time windows) or any specific operator. It is an empty shell waiting to be configured.
    *   **`core/operators/`:** This is the **library of building blocks**. It is a collection of Python packages, where each sub-directory (`crossover/`, `mutation/`, `selection/`) represents a category of operator. Inside each, an `operators.py` file contains the actual functions that implement specific behaviors (e.g., `route_based_crossover`, `relocate_mutation`). These functions are generic and rely on the `instance` object passed to them to check for feasibility.

2.  **The `experiments/` Directory (The Problem Definitions):** This directory contains specific applications of the framework. Each sub-directory (`alg1_cvrp/`, `alg3_vrptwc/`, etc.) defines a unique VRP variant.
    *   **`instance.py` (e.g., `cvrp_instance.py`, `vrptw_instance.py`):** This is the most critical problem-specific file. The `Instance` class within it encapsulates **all knowledge about a specific VRP variant**. It stores the problem data (coordinates, demands, time windows) and provides essential methods like `is_feasible()`, `total_cost()`, and `cheapest_feasible_insertion()`. The `GAFramework` uses these methods to guide its search without needing to know *why* a solution is feasible or not.
    *   **`config.yaml`:** This is the high-level "recipe" for creating a new algorithm. It declaratively defines two things: the problem data for the `Instance` class, and, crucially, **which operators to select from the `core/operators/` library** to solve it.
    *   **`main.py`:** This is the entry point or "wiring" script for an experiment. Its job is to read the `config.yaml`, create the appropriate `Instance` object, create the generic `GAFramework` object, and then pass the instance and configuration parameters to the framework to start the run.

### III. The Complete Algorithm Workflow (Step-by-Step)

This is the sequence of events when you execute a command like `python3 -m experiments.alg3_vrptwc.main`.

**Phase 1: Setup and Initialization**

1.  **Execution as a Module:** The `python3 -m` command tells Python to run the code as a module within a package. This is critical because it makes the entire project structure (including the `core` directory) visible and allows for clean relative imports (like `from .vrptw_instance import ...`).

2.  **Project Path Configuration:** The first thing the `main.py` script does is programmatically find the project's root directory and add it to Python's system path. This guarantees that the `import core.ga_framework` statement will work, regardless of where you run the command from.

3.  **Configuration Loading:** The `main.py` script locates and loads its corresponding `config.yaml` file. This file contains all the parameters for this specific experimental run.

4.  **Problem Instance Creation:** The `main.py` script reads the `instance` and `fleet` sections of the configuration. It uses this data to create a specific instance object (e.g., `VRPTWInstance(...)`). This object now holds all the data and constraint-checking logic for the VRPTWC problem.

**Phase 2: Algorithm "Design" (The Hyper-Heuristic Step)**

5.  **GA Framework Instantiation:** The `main.py` script now creates an instance of the generic `GAFramework`, passing it two things: the problem-specific `instance` object created in the previous step, and the `parameters` section from the configuration.

6.  **Dynamic Operator Selection:** This is the key hyper-heuristic step. Inside the `GAFramework`'s `__init__` method:
    *   It looks at the `operators` dictionary from the passed-in parameters.
    *   For `crossover`, it reads the string `"route_based"`. It then assigns the actual `route_based_crossover` function (which it imported from the `core.operators.crossover` package) to an internal variable, `self.crossover_op`.
    *   For `mutation`, it reads the list `["relocate", "swap"]`. It iterates through this list and appends the actual `relocate_mutation` and `swap_mutation` functions to an internal list, `self.mutation_ops`.
    *   At the end of this step, the generic `GAFramework` object has been **configured and specialized**. It has effectively "designed" a new algorithm on the fly, tailored to the instructions in the `config.yaml`.

**Phase 3: Execution of the Generated Algorithm**

7.  **Starting the Run:** The `main.py` script calls the `ga.run()` method.

8.  **Population Initialization:** The `GAFramework` creates an initial population. To do this, it uses a greedy heuristic that repeatedly calls the **instance's** `cheapest_feasible_insertion()` method. This ensures the initial solutions are created in a way that respects the specific constraints of the problem (e.g., capacity and time windows for VRPTWC).

9.  **The Generational Loop:** The framework now iterates for the specified number of generations. In each generation:
    *   A new population is created.
    *   **Selection:** Individuals are selected for breeding by calling the `tournament_selection` function from the `core.operators.selection` package.
    *   **Crossover:** The framework calls the crossover function it configured earlier (`self.crossover_op`). This function, in turn, uses the **instance's** `cheapest_feasible_insertion()` method to build a valid child.
    *   **Mutation:** The framework **randomly chooses one** of the mutation functions from its configured list (`self.mutation_ops`) and applies it to the child. This introduces variation.
    *   **Evaluation and Repair:** The framework evaluates the new child's fitness by calling the **instance's** `total_cost()` method. If the child is invalid, it calls the **instance's** `is_feasible()` method and may attempt a repair, again using the instance's insertion logic.

10. **Termination and Output:** After the final generation, the `run()` method returns the best solution (chromosome) and its cost to the `main.py` script. The `main.py` script then uses the instance's helper methods (e.g., `render_routes_original_ids`) to format the solution and prints the final results to the console.
